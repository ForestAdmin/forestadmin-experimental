# Migration Note - RPC Cross-Stack Compatibility

## Affected Packages

| Package                                    | Version   |
| ------------------------------------------ | --------- |
| `@forestadmin-experimental/rpc-agent`      | `>2.7.0`  |
| `@forestadmin-experimental/datasource-rpc` | `>1.10.0` |

## Breaking Changes

### 1. SSE (Server-Sent Events) Removal

The schema change detection mechanism via SSE (`eventsource`) has been **replaced by HTTP polling**. The `eventsource` dependency has been removed from `datasource-rpc`.

**Before:** The client connected via SSE to `/forest/rpc-sse` to receive real-time schema change notifications.

**After:** The client periodically polls `/forest/rpc-schema` with an `if-none-match` header (ETag). If the schema hasn't changed, the server responds with `304 Not Modified`.

**Polling configuration:**

```ts
createRpcDataSource({
  uri: 'http://localhost:3352',
  authSecret: process.env.FOREST_AUTH_SECRET,
  pollingInterval: 600, // in seconds (default: 600, min: 1, max: 3600)
});
```

## Nouvelles fonctionnalitĂ©s

### 1. Aggregation Capabilities Support

The RPC schema now carries **aggregation capabilities** for each collection:

- `support_groups`: whether the collection supports grouping
- `supported_date_operations`: list of supported date operations

This allows `datasource-rpc` to correctly configure `setAggregationCapabilities()` on each collection, which is required for charts and aggregations to work properly on the consumer side.

### 2. Native Queries Support

New `/forest/rpc-native-query` endpoint on the `rpc-agent` side, and `datasource-rpc` now exposes `executeNativeQuery()` which forwards calls to the RPC agent.

The introspection schema contains the list of available `native_query_connections`.

### 3. Introspection Fallback

`createRpcDataSource` now accepts an `introspection` option that serves as a **fallback** if the RPC agent is unreachable at startup:

```ts
createRpcDataSource({
  uri: 'http://localhost:3352',
  authSecret: process.env.FOREST_AUTH_SECRET,
  introspection: cachedSchema, // optional, used if the server is down
});
```

Please note that introspection files have changed to use those generated by rpc agents.

## `rpc-agent` - Key Points

### Schema Serialization

The `RpcAgent` builds and caches the schema at startup (`buildSchema`). The schema is:

- Serialized in **snake_case** (RPC protocol convention)
- Hashed with SHA-1 to generate an **ETag**
- Saved to `.forestadmin-rpc-schema.json`
- **Not sent** to Forest Admin (the RPC agent does not communicate directly with Forest Admin)

### Marking Collections as RPC

Collections that belong to the consumer (not the RPC agent) can be marked:

```ts
agent.addDataSource(factory, { markCollectionsAsRpc: true });
// or
agent.markCollectionsAsRpc('CollectionA', 'CollectionB');
```

Relations between RPC collections and local collections are extracted into `rpc_relations` in the schema, then reconciled on the consumer side via the `reconciliateRpc` plugin.

## `datasource-rpc` - Key Points

### Basic Usage

```ts
const { createAgent } = require('@forestadmin/agent');
const {
  createRpcDataSource,
  reconciliateRpc,
} = require('@forestadmin-experimental/datasource-rpc');

const agent = createAgent(options).addDataSource(
  createRpcDataSource({
    uri: 'http://localhost:3352',
    authSecret: process.env.FOREST_AUTH_SECRET,
  }),
);

// If the agent also has local datasources with cross-stack relations
agent.use(reconciliateRpc);
```

### `reconciliateRpc` Plugin

This plugin is **required** if relations exist between the RPC agent's collections and the consumer's local collections. It:

1. Disables search on non-searchable collections
2. Recreates relations (ManyToOne, OneToMany, OneToOne, ManyToMany) between cross-stack collections

It supports a `rename` option to handle cases where collections are renamed:

```ts
agent.use(reconciliateRpc, { rename: { OldName: 'NewName' } });
// or with a function
agent.use(reconciliateRpc, { rename: name => `prefix_${name}` });
```

## Authentication Protocol

All RPC requests are signed via HMAC-SHA256:

- Header `X_SIGNATURE`: HMAC of the timestamp with the `authSecret`
- Header `X_TIMESTAMP`: ISO timestamp of the request
- Header `forest_caller`: caller information (serialized as JSON)

The `authSecret` must be **identical** on both the `rpc-agent` and `datasource-rpc` sides.
